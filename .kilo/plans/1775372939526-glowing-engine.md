Patch the social/media query builders to prefer canonical artist + clean album/project context, avoid item-title/package metadata when album context exists, and keep YouTube `maxResults` at 50 while expanding the deduped candidate pool for post-collection ranking.

Scope

- Update only `src/validation/providers/query-utils.ts` and `src/validation/providers/social.ts`.
- Do not change sold/browse query builders.
- Do not change YouTube ranking heuristics beyond feeding them a better candidate set.

Implementation plan

1. Add a dedicated social/media album-context helper in `src/validation/providers/query-utils.ts`.
   - Introduce a helper that resolves the cleanest social-facing album/project phrase from `request.item.relatedAlbums` first.
   - Prefer the first related album whose normalized phrase still has semantic tokens after stripping conversation noise.
   - If no related album survives cleanup, fall back to the existing title-derived album phrase path.
   - Keep this helper narrow and reusable so it can later switch cleanly to a canonical album schema without changing the Twitter/YouTube/Reddit builders.

2. Route Twitter, YouTube, and Reddit query builders through that clean album/project helper.
   - `buildTwitterQueryPlan` should generate album/project-led conversation queries first, e.g. `artist + album`, quoted variant, then lighter fallbacks only if album context is absent.
   - `buildYouTubeQueryPlan` should generate `artist + album`, `artist + album + official`, `artist + album + mv`, `artist + album + music video`, `artist + album + teaser` from the clean album/project phrase.
   - `buildRedditQueryPlan` should also use the clean album/project phrase rather than raw `albumPhrase`/title contamination.
   - Avoid promoting raw listing-title or version/package metadata when the helper has album/project context available.

3. Preserve the current YouTube per-query API fetch size at `YOUTUBE_SEARCH_MAX_RESULTS = 50`.
   - Do not reduce or otherwise regress this constant.
   - Keep existing `maxResults: YOUTUBE_SEARCH_MAX_RESULTS` behavior in the search request.

4. Increase the deduped YouTube candidate pool after search collection in `src/validation/providers/social.ts`.
   - Raise `YOUTUBE_MAX_CANDIDATE_VIDEOS` above 50 so the collector can retain more unique video IDs across all query variants.
   - Keep collection broad, then continue using the existing strict ranking/sort pipeline to choose the final candidate.
   - Ensure the larger pool still fits the YouTube videos details request limit strategy; if needed, cap to a safe single-request batch size that is larger than current behavior without altering ranking rules.

5. Keep diagnostics aligned with the new query source.
   - Preserve `family`, `queryCandidates`, `selectedQuery`, and candidate debugging outputs.
   - Ensure debug output reflects the album/project-centric queries so future canonical album integration is observable.
