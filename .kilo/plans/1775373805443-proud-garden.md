Implement Airtable-resolved query-context precedence in the validation pipeline, fix the likely browse-provider baseUrl issue, and limit lint cleanup to files touched by this patch.

Scope

- Update only validation request/schema/types and validation provider files involved in query planning/debug, plus any directly related baseUrl fix in the validation browse provider path.
- Do not remove existing provider query builders; keep them as fallback.
- Do not attempt repo-wide lint cleanup; only address lint issues introduced or exposed in touched files.

Discoveries

- `ValidationRunRequest` currently has no `queryContext` field in `src/validation/types.ts`, and `validationRunRequestSchema` in `src/validation/schemas.ts` does not accept one.
- All current provider query inputs are still generated internally from `src/validation/providers/query-utils.ts`:
  - browse: `buildBrowseQueryPlan(...)`
  - sold: `buildSoldQueryPlan(...)`
  - social: `buildTwitterQueryPlan(...)`, `buildYouTubeQueryPlan(...)`, `buildRedditQueryPlan(...)`
  - stubbed Terapeak: `buildValidationQueryCandidates(...)`
- `runValidation()` in `src/validation/run-validation.ts` passes the raw request to each provider and then assembles top-level debug. This is the best orchestration point for surfacing request-level query-context debug.
- Existing provider debug already includes `queryCandidates`, `selectedQuery`, and diagnostics. This should be extended rather than replaced.
- There are no validation-specific lint/check blockers in current `npm run check` output; repo-wide output is warnings-only and mostly unrelated. Touched-files-only cleanup is appropriate.
- The likely “baseUrl error” source is `src/validation/providers/ebay.ts`, which composes a browse URL via `getBaseUrl(environment)` and calls `getWithFullUrl(...)`. No equivalent baseUrl plumbing exists in other validation providers.

Implementation plan

1. Extend the validation request model to accept Airtable query context.
   - In `src/validation/types.ts`, add an optional `queryContext` object to `ValidationRunRequest` with these nullable string/boolean fields:
     - `validationScope`
     - `queryScope`
     - `directSearchQuery`
     - `directQueryActive`
     - `resolvedSearchArtist`
     - `resolvedSearchItem`
     - `resolvedSearchEvent`
     - `resolvedSearchLocation`
     - `resolvedSearchQuery`
   - In `src/validation/schemas.ts`, mirror that structure as an optional schema so older callers remain valid.

2. Add a small shared query-context resolution layer in `src/validation/providers/query-utils.ts`.
   - Introduce a shared provider debug/resolution type export for query sourcing, such as:
     - `queryContextUsed`
     - `querySource: 'airtable_resolved' | 'provider_builder'`
     - `resolvedSearchQuery`
     - `validationScope`
     - `queryScope`
     - optional `fallbackReason`
   - Add helpers to:
     - sanitize the Airtable-resolved query string
     - treat blank values as unusable
     - reject strings beginning with `error:` case-insensitively
     - build a merged query plan where a valid Airtable query is prepended ahead of existing provider-generated candidates
   - Keep provider-specific fallback builders intact; the override layer should wrap them, not replace them.

3. Update browse/sold/social/Terapeak query planning to prefer the Airtable-resolved query.
   - In `src/validation/providers/ebay.ts`, route `buildEbayValidationQueries(...)` and `getEbayValidationSignals(...)` through the new merged plan helper:
     - valid Airtable query first
     - existing browse query plan afterward as fallback
   - In `src/validation/providers/ebay-sold.ts`, do the same with sold query candidates.
   - In `src/validation/providers/social.ts`, do the same independently for Twitter, YouTube, and Reddit query plans:
     - valid Airtable query first
     - existing provider-specific conversation/discussion/media fallbacks afterward
   - In `src/validation/providers/terapeak.ts`, align the stub debug queries with the same resolved-query-first behavior so diagnostics stay consistent across providers.

4. Preserve provider flexibility while enforcing Airtable precedence.
   - Use this precedence everywhere:
     1. valid `validation.queryContext.resolvedSearchQuery`
     2. existing provider builder plan
   - Do not pass `error: ...` strings to provider APIs.
   - If the Airtable query is invalid/error-prefixed, record that in debug and fall back to the provider builder plan.
   - For direct query override (`directQueryActive = true`), the resolved query still becomes the first provider candidate; existing provider fallbacks remain behind it for resilience.

5. Add clear debug output proving which query source was used.
   - In `src/validation/types.ts`, extend provider debug-bearing signal types to include the shared query-resolution metadata without breaking current fields.
   - In `src/validation/run-validation.ts`, enrich top-level debug with request-level query-context visibility and preserve per-provider debug.
   - In provider outputs, add the requested fields or equivalent nested metadata:
     - `queryContextUsed`
     - `resolvedSearchQuery`
     - `validationScope`
     - `queryScope`
     - `querySource`
   - Preserve existing fields like `family`, `queryCandidates`, `selectedQuery`, `queryDiagnostics`, and YouTube candidate debug.

6. Fix the likely browse-provider baseUrl issue in `src/validation/providers/ebay.ts`.
   - Inspect how `getBaseUrl(environment)` is being used for Browse and adjust the URL composition so `getWithFullUrl(...)` receives the correct absolute Browse endpoint.
   - Keep the fix minimal and local to the validation browse path unless inspection reveals a shared helper issue.
   - Clear any lint/style issues introduced in the touched validation/baseUrl files while making this change.

7. Verify behavior with targeted checks.
   - Run build/tests/check after implementation.
   - Confirm these cases conceptually or with targeted tests if added:
     - Direct query override: `resolvedSearchQuery = money` becomes the primary provider input.
     - Album scope: `resolvedSearchQuery = bts arirang` appears first in provider debug, not old title-derived variants.
     - Error-prefixed query: `resolvedSearchQuery = error: ...` is not sent to providers and fallback builder debug is visible.
     - Browse provider still executes against the corrected absolute eBay Browse URL.

Notes for implementation

- Prefer a shared merged-plan helper in `query-utils.ts` rather than duplicating override validation in each provider.
- Keep the existing social album/project cleanup work intact; Airtable-resolved queries should override only when present and valid.
- Preserve backward compatibility by making `queryContext` optional end-to-end.
- If no validation tests exist, keep the code changes narrowly typed and rely on existing build/test coverage plus targeted debug inspection.
